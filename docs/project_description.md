# Платформа помощи гражданам — описание проекта

## 1. Описание проекта

**Цель:** сервис для приёма и обработки обращений граждан по городским проблемам (ямы на дорогах, освещение, ЖКХ и т.п.). Гражданин создаёт заявку через Telegram-бота, оператор назначает исполнителя и контролирует выполнение, исполнитель выполняет работу и прикладывает доказательства (фото и т.д.), оператор подтверждает или отклоняет их. После выполнения гражданин может поставить оценку и оставить отзыв.

**Пользователи системы:**
- **Гражданин** — создаёт заявки, получает уведомления об изменении статуса, в конце может оценить работу и оставить отзыв.
- **Оператор** — принимает заявки, назначает исполнителей, проверяет доказательства выполнения (пруфы), выступает «горячей линией» по заявкам.
- **Исполнитель** — выполняет заявки, меняет статусы, прикладывает пруфы (фото и т.д.).
- **Суперпользователь (админ)** — создаёт операторов и исполнителей, управляет системой.

**Польза:** упорядоченный учёт обращений, прозрачные статусы, возможность оценки качества работы и обратная связь от граждан.

## 2. Пользовательские сценарии (User Cases)

### Гражданин
1. **Регистрация в боте** — открывает бота в Telegram, нажимает «Старт»; при необходимости система связывает его с учётной записью (по Telegram ID).
2. **Создание заявки** — выбирает «Создать заявку», вводит краткое название и описание (адрес, суть проблемы); заявка создаётся и получает номер и статус «Создана».
3. **Просмотр статуса** — через команду «Мои заявки» видит список своих заявок и их статусы (создана, в работе, на проверке, выполнена и т.д.).
4. **Получение уведомлений** — при изменении статуса или при подтверждении/отклонении пруфов получает уведомление в Telegram.
5. **Оценка и отзыв** — когда заявка выполнена, может выставить оценку (например, от 1 до 5) и/или подтвердить выполнение и оставить текстовый отзыв.

### Оператор
1. **Вход по Telegram** — при первом запуске бота привязывается к учётной записи оператора (по Telegram ID, создан заранее суперюзером/админом).
2. **Просмотр заявок** — видит заявки, назначенные ему, и полную информацию по каждой (описание, адрес, исполнитель, пруфы).
3. **Назначение исполнителя** — выбирает заявку и назначает на неё исполнителя; заявка переходит в работу.
4. **Просмотр и решение по пруфам** — когда исполнитель прикладывает доказательства выполнения, оператор видит их и подтверждает или отклоняет; при изменении статуса заявки оператор получает уведомления.
5. **Уведомления** — получает в Telegram сообщения о новых заявках, запросах на изменение и необходимости проверить пруфы.

### Исполнитель
1. **Вход по Telegram** — при первом запуске бота привязывается к учётной записи исполнителя (создан оператором/суперюзером с указанием Telegram ID).
2. **Просмотр назначенных заявок** — видит список заявок, назначенных ему.
3. **Смена статуса** — отмечает заявку «В работе», при завершении — «На проверке» или «Выполнена» в зависимости от правил системы.
4. **Загрузка пруфов** — прикладывает к заявке файлы (фото и т.д.) и краткий комментарий; оператор затем подтверждает или отклоняет пруфы.

### Суперюзер / админ
1. **Создание операторов и исполнителей** — через админ-панель или скрипт создаёт пользователя с ролью «оператор» или «исполнитель», указывает логин, пароль и Telegram ID; при первом входе в бота пользователь привязывается по этому ID.
2. **Создание суперпользователя** — через скрипт создаётся ещё один суперюзер с логином, паролем и Telegram ID.
3. **Управление данными** — в админ-панели может просматривать и при необходимости править пользователей, заявки, типы заявок и пруфы (CRUD).

## 3. Глоссарий (сложные термины — простыми словами)

- **API** — способ, по которому одна программа запрашивает данные или действия у другой по сети. Например, бот может «спросить» у сервера список заявок через API.

- **Бэкенд** — часть системы, которая работает на сервере: хранит данные, проверяет права, обрабатывает запросы. В нашем проекте это приложение на FastAPI и база данных.

- **Фронт (фронтенд)** — то, с чем напрямую взаимодействует пользователь: кнопки, экраны, формы. Сейчас фронт у нас — это Telegram-бот; в будущем им может стать веб-сайт или мобильное приложение, которые будут обращаться к тому же бэкенду через API.

- **Telegram Bot (Telegram-бот)** — программа, которая общается с пользователем внутри мессенджера Telegram. Пользователь пишет боту команды и сообщения, бот отвечает и может отправлять уведомления.

- **Заявка** — одно обращение гражданина по конкретной проблеме (например, «Яма на улице X»). У заявки есть номер, описание, статус, назначенный оператор и исполнитель.

- **Статус заявки** — этап жизни заявки: создана, в работе, на проверке пруфов, выполнена, отклонена. По статусу видно, на каком этапе находится заявка.

- **Пруф (доказательство)** — файл (фото и т.п.) или ссылка, которую исполнитель прикладывает к заявке как доказательство выполненной работы. Оператор проверяет пруф и подтверждает или отклоняет его.

- **Оператор** — сотрудник, который принимает заявки, назначает исполнителей и проверяет пруфы. Он видит полную информацию по заявкам и выступает «горячей линией» для граждан и исполнителей.

- **Исполнитель** — сотрудник, который непосредственно выполняет работу по заявке (например, ремонт) и прикладывает пруфы. Создаётся оператором или суперюзером.

- **Суперюзер** — пользователь с максимальными правами: может создавать операторов и исполнителей, управлять данными через админ-панель. Обязательно привязан к Telegram для входа в бота.

- **Роль** — тип пользователя в системе: гражданин, оператор, исполнитель или суперюзер. От роли зависят доступные действия (например, только оператор может подтверждать пруфы).

- **База данных** — хранилище структурированных данных (пользователи, заявки, пруфы и т.д.). В проекте используется PostgreSQL: надёжная база, которая хранит всё даже после перезапуска сервера.

- **Docker** — технология для «упаковки» приложения и его окружения (например, база данных) в контейнеры. Позволяет поднять весь проект одной командой (docker-compose) на любом компьютере без ручной установки PostgreSQL и версий Python.

- **Swagger** — встроенная в FastAPI интерактивная документация по API: можно открыть страницу в браузере и посмотреть все запросы (эндпоинты), параметры и ответы, а также отправить тестовый запрос.

- **Админка (админ-панель)** — веб-интерфейс для администратора: просмотр и редактирование пользователей, заявок, типов заявок и пруфов без написания кода. В проекте реализована через SQLAdmin по адресу /admin.

- **Зоны ответственности** — задумка на будущее: категории заявок по типу проблемы или району (например, «Дороги», «ЖКХ»). В текущей версии (MVP) реализована как заглушка: есть типы заявок, но без сложной логики распределения.

- **MVP (Minimum Viable Product)** — минимально жизнеспособный продукт: версия системы с основным набором функций, достаточным для демонстрации и курсовой работы, без части «на будущее» (например, полноценный вход через API с привязкой к Telegram пока не реализован).
